bns / scanner — Dashboard v0

Design reference for the operator-facing dashboard. Three views: the explorer (main working surface), the command palette (power-user navigation), and the node detail drawer (investigate a single node). OSINT terminal aesthetic — data-dense, monospace-first, Bitcoin orange reserved for brand and CTAs.

v0 · draft · ivan@hacknodes.lab

Design notes
  • The expanded row is the product — without it this is "another bitnodes". Everything else should make it easier to reach.
  • Query bar is a filter builder, not a text search. Saved queries get first-class treatment.
  • Stats strip shows deltas vs last hour. Never absolute-only — context matters more than the number.
  • Map is intentionally absent from this view. It goes in a toggle or secondary tab.

Design notes
  • Ranking is context-aware. "exp" surfaces the last run query first, not the alphabetically-first "exp" item.
  • The palette is the CLI. Every entry here must have a CLI form and a REST form with matching grammar.
  • L402 is a first-class action, visually distinct (orange icon, sat amount), but not intrusive — it appears when there's something pay-gated in the current context.
  • Consider a preview pane on ⇥ (Tab) — hover-preview what an action will do before committing.
03VIEW

Opens when a row is expanded or a node is opened directly. The dimmed sliver on the left keeps context of the surrounding list. Tabs prioritize exposure when findings exist, overview otherwise. Footer holds safe actions first; L402 unlock sits on the right as the premium action.

Design notes
  • Default tab depends on state. When exposure findings exist, exposure opens first; otherwise overview.
  • Red badges are reserved for actionable findings. Neutral counts (peers 14) stay gray — never mix.
  • Free tier shows the what, paid unlocks the how. PoCs, exploit scripts, temporal correlation with similar nodes in the same ASN, concrete patch recommendations.
  • The dimmed sliver keeps list context visible — useful when doing a sweep across many nodes.
04OPEN QUESTIONS

Still to decide

Decisions we paused on. Each has implications for implementation and UX testing.

Q1 · LAYOUT
Drawer expansion: inline vs. side panel
Inline expansion keeps list context but gets cramped with many findings. A right-side drawer gives space but hides the list. Hybrid: brief inline summary, full drawer on "open full report". Needs prototyping against real data.
Q2 · INFORMATION
Map placement
Out of v0 entirely, or subordinate sidebar? Operators say they want it but don't use it day-to-day. Ship without, add in v1 if retention data shows demand.
Q3 · SEMANTICS
Coloring ASNs by risk
Tempting to color cheap-hosting ASNs amber and privacy-provider ASNs differently. Risks over-saturation and false signals. Start with a single badge (ASN name only) and evolve if user research demands it.
Q4 · DELIVERY
Alerts channel
In-product toast, email, Nostr DM, webhook, Slack? Nostr DM is the brand-aligned option and technically easy; webhooks open enterprise use cases. Email as a floor. Start with Nostr + webhook for v0.
Q5 · MONETIZATION
What goes behind the L402 paywall
Proposal: free tier answers "is there a problem" — paid tier answers "how do I fix it or use it". PoCs, patch recommendations, historical correlation, exportable reports. Needs a line-item breakdown validated with 3–5 potential customers before shipping.
Q6 · ACCESSIBILITY
Light mode?
Hacker aesthetic is dark-first, but some operators work in bright offices or share screens. Build with CSS custom properties so a light mode is possible later without refactoring. Not a v0 deliverable.
